05 · Safety architecture
Writer ≠ checker

Defensa multicapa

El modelo genera texto. El código conserva autoridad.

L0

Determinístico

Regex, fixtures T1–T7, crisis y abuso con sustitución.

L1–L2

Moderación

Entrada y salida; cascada fail-closed cheapest-first.

L3

Precedencia

Código puro elige plantilla, reemplazo o respuesta normal.

L4

Humano

Revisión de fixtures, templates, recursos y releases.

Fail-closed: si una capa no puede concluir, no se degrada el piso de seguridad.

Evidencia requerida

  • Suite crítica versionada al 100%.
  • Reporte adversarial independiente.
  • FPR/FNR por categoría y edad.
  • Chaos test de alerta y callback.

07 · Threat model
Trust boundaries y controles

Amenazas que diseñan la solución

No protegemos “un chat”. Protegemos límites de confianza.

Entrada hostil

Prompt injection / jailbreak

Precedencia determinística, contexto revisado, moderación de salida y tests adversariales.

Acceso indebido

IDOR / tutela

Autorización por relación tutor-menor, ownership checks y tests cross-user.

Datos

PII / exfiltración

Redacción, minimización, allowlist, logs sin texto y acceso nominativo.

Operación

Alertas perdidas

Outbox, idempotencia, dedupe, retry y reconciliación de pendientes.

Contenido

Poisoning / vencimiento

Provenance, hash, owner, revisión, expiración y separación train/eval.

Supply chain

Model swap / dependencia

Lockfile, SBOM, model ID, hashes, canary y rollback.
